import os
from datetime import datetime

CACHE_DIR = os.path.expanduser("~/.hermes/profiles/sales/cache")

def _format_contacts_machine(contacts_data):
    """
    格式化为 handler 可直接发送给查询人的结构化文本。
    格式与现有 contact_details 成功案例一致：
      【业主】公司名\n姓名 | 手机号 | 角色 | 职责\n...
    无 emoji，无终端装饰字符。
    """
    if not contacts_data or not contacts_data.get('enterprises'):
        return ''
    lines = []
    for ent in contacts_data['enterprises']:
        company = ent.get('company', '')
        category = ent.get('category', '')
        header = f"【{category}】{company}" if category else f"【】{company}"
        lines.append(header)
        for c in ent.get('contacts', []):
            name = c.get('name') or '???'
            phone = c.get('phone') or '???'
            dept = c.get('department', '')
            pos = c.get('position', '')
            duty = c.get('duty1', '')
            role = f"{dept}{pos}".strip()
            line = f"{name} | {phone}"
            if role:
                line += f" | {role}"
            if duty:
                line += f" | {duty}"
            lines.append(line)
        lines.append('')
    return '\n'.join(lines).strip()

def _write_contacts_cache(project_id, contacts_data):
    """将联系人查询结果写入缓存文件"""
    os.makedirs(CACHE_DIR, exist_ok=True)
    os.chmod(CACHE_DIR, 0o700)
    
    cache_file = os.path.join(CACHE_DIR, f"contacts_{project_id}.json")
    
    cache = {
        "project_id": project_id,
        "queried_at": datetime.now().isoformat(),
        "contact_count": contacts_data.get('contact_count', 0),
        "contacts_machine": _format_contacts_machine(contacts_data),
    }
    
    # 原子写入：先写临时文件，再 rename
    tmp = cache_file + ".tmp"
    with open(tmp, 'w', encoding='utf-8') as f:
        json.dump(cache, f, ensure_ascii=False, indent=2)
    os.replace(tmp, cache_file)
    os.chmod(cache_file, 0o600)

#### 改动 3.2: 新增 `_read_contact_details()` 函数

**位置**: L119（`_is_placeholder_text()` 之后），新函数

**描述**: 三级优先级读取联系人明细。返回 `(text, is_valid, invalid_reason, source)` 元组。

**完整逻辑**:
```python
def _read_contact_details(matched):
    """
    三级优先级读取联系人明细。
    
    优先级：
      1. artifact 缓存文件（contact_artifact_path → contacts_machine）
      2. pending 记录中的 contact_details 字段
      3. 均为占位 → BLOCKED
    
    Returns:
        (text, is_valid, invalid_reason, source)
        source ∈ {"artifact_cache", "pending_record", None}
    """
    # 优先级1: artifact 缓存文件
    artifact_path = matched.get("contact_artifact_path")
    if artifact_path:
        # 支持相对路径（相对于 CONTACTS_CACHE_DIR）和绝对路径
        if os.path.isabs(artifact_path):
            full_path = artifact_path
        else:
            full_path = os.path.join(CONTACTS_CACHE_DIR, artifact_path)
        
        if os.path.exists(full_path):
            try:
                with open(full_path, 'r', encoding='utf-8') as f:
                    cache = json.load(f)
                text = cache.get("contacts_machine", "") or cache.get("contacts_text", "")
                if text:
                    is_ph, reason = _is_placeholder_text(text)
                    if not is_ph:
                        return text, True, None, "artifact_cache"
                    else:
                        return text, False, f"artifact_cache_is_placeholder:{reason}", None
            except (json.JSONDecodeError, IOError, KeyError):
                pass  # 缓存损坏，回退到下一级
    
    # 优先级2: pending 记录中的 contact_details
    contact_details = matched.get("contact_details", "")
    if contact_details:
        is_ph, reason = _is_placeholder_text(contact_details)
        if not is_ph:
            return contact_details, True, None, "pending_record"
        return contact_details, False, reason or "pending_record_placeholder", "pending_record"
    
    # 优先级3: 完全无数据
    return "", False, "no_contact_data_available", None
```

#### 改动 4.2: 新增 artifact/cache 流程说明

**位置**: L182（数据流全景图之后），新小节

**描述**: 新增一段解释 artifact/cache 机制，让模型理解为什么不再需要手动传联系人数据。

**改动后** (插入在原有数据流全景图之后):
```
**🔴 v2.x 更新：artifact/cache 机制（数据流优化）**

**为什么不再需要手动传 contact_details：**

之前你需要在 CLI 命令中嵌入完整的联系人原文，但实测发现模型在多轮对话后容易传入占位文本（如"已获取22位联系人，待审批后输出"），导致审批通过后无法送达。

**新机制：** \`ccpc360_query.py contacts\` 执行时会自动将完整联系人数据写入缓存文件：
\`\`\`
~/.hermes/profiles/sales/cache/contacts_{项目ID}.json
\`\`\`

**你只需要：**
1. 执行 \`ccpc360_query.py contacts {项目ID}\` 查询联系人（照旧）
2. 调用 \`pending_approval.py add ... --contact-cache-from {项目ID}\`（用项目ID引用缓存）
3. 脚本自动从缓存文件读取完整联系人数据，**你不需要在命令中嵌入任何联系人详情**

**数据流（v2.x）：**
\`\`\`
步骤1: ccpc360_query.py contacts {项目ID}
        → 打印联系人到 stdout（你可以看到）
        → 自动写缓存到 cache/contacts_{项目ID}.json（脚本自动完成）

- 之前你需要在 CLI 参数中传完整联系人 → 容易出错

- \`contact_details\` 仍可作为 fallback（缓存文件不存在时使用），正常情况传空字符串 \`""\` 即可
```

## 二、artifact/cache 写入逻辑

### 1. 写入时机

在 `ccpc360_query.py` 的两个位置：
- `cmd_contacts()`: `get_project_contacts()` 成功返回 → 打印输出 → **写入缓存** → 返回
- `cmd_full()`: `get_project_contacts(pid)` 成功返回 → **写入缓存** → 打印输出 → 返回

**仅在解密模式（decrypt=True，默认）下写入**。`--no-decrypt` 模式跳过写入。

### 2. 缓存文件结构 (JSON Schema)

```json
{
  "project_id": "1855768437552332802",
  "queried_at": "2026-05-12T15:00:00.000000",
  "contact_count": 22,
  "contacts_machine": "【业主】广东三津食品有限公司\n彭苏城 | 15323517606 | 项目知情人\n方增满 | 13622350255 | 法人 | 项目参与人\n...\n\n【设计】广东鸿宇建筑与工程设计顾问有限公司\n叶仲彬 | 13751333468 | 老板 | 项目统筹\n..."
}
```

**字段说明**:
| 字段 | 类型 | 用途 |
|------|------|------|
| `project_id` | string | 项目 ID（用于校验和调试） |
| `queried_at` | ISO datetime | 查询时间（调试和过期判断） |
| `contact_count` | int | 联系人总数（校验用） |
| `contacts_machine` | string | **核心字段**。结构化联系人文本，格式与 handler 可直接发送的格式一致。`【企业类型】公司名\n姓名 \| 手机号 \| 角色 \| 职责\n...` |

**不需要 `contacts_raw`**: 原始 API 响应含大量冗余字段（加密值、解密中间态、坐标等），对 handler 无意义。只保留格式化后的 `contacts_machine`。

**不需要 sha256 校验**: 缓存文件由脚本原子写入（先 tmp 再 rename），JSON 解析自然校验完整性。文件包含手机号但本地位于 700 权限目录，安全边界足够。

### 3. 缓存文件路径生成

```python
safe_filename = f"contacts_{project_id}.json"
# project_id 为纯数字字符串（如 "1855768437552332802"），天然安全
# 无需额外 sanitize
full_path = os.path.join(CACHE_DIR, safe_filename)
# → ~/.hermes/profiles/sales/cache/contacts_1855768437552332802.json
```

### 4. 文件权限

```python
os.chmod(CACHE_DIR, 0o700)       # 目录：仅 owner 可访问
os.chmod(cache_file, 0o600)      # 文件：仅 owner 可读写
```

### 5. 原子写入

使用 `tmp + rename` 策略，防止并发读取不完整文件：
```python
tmp = cache_file + ".tmp"
with open(tmp, 'w') as f:
    json.dump(cache, f, ...)
os.replace(tmp, cache_file)  # 原子操作（POSIX）
```

## 四、handler.py 变更

### 1. `_read_contact_details()` 完整逻辑

```
function _read_contact_details(matched):
    
    # === 优先级1: artifact 缓存文件 ===
    artifact_path = matched.get("contact_artifact_path")
    if artifact_path is not None and not empty:
        full_path = resolve(artifact_path)
        if file_exists(full_path):
            try:
                cache = json.parse(read_file(full_path))
                text = cache.contacts_machine or cache.contacts_text or ""
                if text is not empty:
                    is_ph, reason = _is_placeholder_text(text)
                    if not is_ph:
                        return (text, True, None, "artifact_cache")
                    else:
                        return (text, False, reason, None)
            catch (JSONDecodeError, IOError, KeyError):
                pass  # 回退到下一级
    
    # === 优先级2: pending 记录中的 contact_details ===
    contact_details = matched.get("contact_details", "")
    if contact_details is not empty:
        is_ph, reason = _is_placeholder_text(contact_details)
        if not is_ph:
            return (contact_details, True, None, "pending_record")
        return (contact_details, False, reason, "pending_record")
    
    # === 优先级3: 完全无数据 ===
    return ("", False, "no_contact_data_available", None)
```

### 2. 三级回退优先级

| 优先级 | 数据源 | 读取方式 | 成功后 source 值 |
|--------|--------|----------|-----------------|
| 1 (最高) | artifact 缓存文件 | `json.load(cache_file)['contacts_machine']` | `artifact_cache` |
| 2 | pending.contact_details | 直接读取字段 | `pending_record` |
| 3 (兜底) | — | — | BLOCKED |

### 3. artifact 文件读取失败的处理

| 失败类型 | 处理 |
|----------|------|
| 文件不存在 | 静默回退到优先级2 |
| JSON 解析失败 | 静默回退到优先级2 |
| contacts_machine 为空 | 尝试 contacts_text，仍为空则回退到优先级2 |
| contacts_machine 为占位文本 | 标记 invalid，不继续回退（文件存在但内容无效 = 明确失败） |

### 4. artifact 文件内容为占位文本的处理

直接返回 `(text, False, reason, None)`，**不继续回退**到优先级2。理由：缓存文件存在意味着 CCPC360 查询已执行成功，文件内容为占位文本是异常状态，应明确报告而非静默降级。

### 5. `_send_approval_result()` 修改点

| 原逻辑 | 新逻辑 |
|--------|--------|
| 从 `matched['contact_details']` 直接读取 | 调用 `_read_contact_details(matched)` |
| 用 `_is_placeholder_text()` 内联检测 | 由 `_read_contact_details()` 统一处理 |
| 用 `matched['contact_details_valid']` 标记辅助判断 | 不再需要（`_read_contact_details()` 自行判断） |
| blocked_msg 显示 `placeholder_reason` | blocked_msg 显示 `invalid_reason` + `source` |

**保留不变**:
- `_is_placeholder_text()` 函数（仍作为最后防线）
- 三级兜底发送 L1/L2/L3 逻辑
- SALES_HOME_CHANNEL 留痕逻辑
- `delivery_status` / `delivery_failure_reason` 写入逻辑

步骤4: SALES 判断权限（SOUL.md 辖区映射表）

   ↓  数据: contact_preview（脱敏，给 CY 看）

步骤7: CY 回复审批关键词
   ↓  触发: hook (handler.py) 在 agent:start 时执行

   ↓  优先级2: 读取 pending.contact_details（fallback）

| 5 | `--contact-cache-from` 的 project_id 与实际缓存文件不匹配 | 中 | 中 | pending_approval.py 打印缓存联系人数量到 stderr（如 "从缓存读取联系人: 22 位"），模型/Human 可校验 |

| 8 | 旧 cache 文件堆积（磁盘） | 低 | 低 | 每个缓存文件约 2-5KB，1000 个项目约 5MB；后续可加 TTL 清理脚本 |
